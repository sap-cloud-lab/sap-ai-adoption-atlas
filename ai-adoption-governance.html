AI Adoption Framework

The answer is rarely SAP-only or custom-only.

Choose the right delivery route for each use case, then govern the portfolio as one hybrid AI workspace.

Compare SAP Business AI, our managed custom AI platform, and a hybrid approach using process fit, permitted SAP access, three-year cost, adaptability, control, and lock-in—not vendor preference.

See a worked example See the hybrid workspace

WHY — Start with the business problem

Define the decision, delay, manual work, or control weakness that must change. If there is no measurable process outcome, do not start an AI programme.

HOW — Choose buy, build, or hybrid

Buy what SAP delivers well, build what must reflect the customer’s operating model, and combine both when access or control boundaries are mixed.

WHERE — Put AI at the safest point

Use only supported interfaces and endorsed agent pathways. Keep policy checks, approval, posting, and reconciliation deterministic and governed.

Does SAP allow customer-built AI agents?

Yes—SAP documents third-party agents, side-by-side applications, A2A, MCP, and customer-built services. The boundary is interface-specific: if a required read or action has no supported doorway in that edition and release, an external agent cannot perform that step supportably.

Use customer-specific AI where it adds advantage. Use SAP as the system of record. Use published APIs, A2A, or MCP as the doorway. Use SAP-native or human action where the doorway is closed.

Three operating models for SAP AI

Make the choice per use case. A hybrid portfolio is the likely enterprise outcome, not a mandatory answer for every individual process.

Scenario 1

Use SAP Business AI Platform

Choose SAP’s packaged route when SAP already delivers the business capability and its embedded controls are worth more than bespoke flexibility.

Open architecture image

Public Edition setup

  1. Confirm Joule entitlement, region, language, tenant release, and the specific AI capability.
  2. Prepare BTP, SAP Build Work Zone, IAS/IPS, and user attributes.
  3. Register the S/4 system and run the Joule booster or create the Joule formation.
  4. Verify SAP_COM_0882 and assign SAP_CORE_BC_UI_JOU_PC when using role-based visibility.

Private Edition setup

  1. Confirm RISE with SAP, SAP-managed data centre, installed release, feature package, and required UI5 level.
  2. Register the system, configure destinations, and include it in the Joule formation.
  3. Validate every capability against the release-specific capability guide—Joule availability does not mean every agent is available.
  4. Test roles, navigation, trust, upgrades, and operational ownership in a non-production landscape.
Scenario 2

Operate our Managed Custom AI Platform

We build and operate the reusable reasoning, orchestration, controls, and SAP adapters—on our managed platform or inside the customer’s BTP environment.

Open architecture image

MCP is useful when

  • Several agent or model hosts need one governed tool catalogue.
  • Integration Suite should expose REST, OData, SOAP, HTTP, or approved RFC tools with policies and monitoring.
  • Joule Studio or another MCP client needs reusable read and draft tools.

MCP is optional when

  • One fixed workflow can use a simpler typed REST/OpenAPI integration.
  • The agent does not need dynamic tool discovery.
  • The extra protocol layer would add more operations than reuse.

MCP never proves permission

A working MCP-to-SAP call is not automatically a permitted architecture. Validate the exact AI use under the SAP API Policy and obtain written confirmation when the documented route is unclear.

Scenario 3

Operate a Hybrid AI Workspace

Choose the delivery route per use case. Use SAP where native context or a closed action is decisive, and our managed platform where customer-specific reasoning and supported access create the better outcome.

Open architecture image
Hybrid is not a compromise. Buy what only SAP can safely deliver; build what reflects the customer’s real process; govern both as one workspace.

SAP enables external AI—but through supported doorways

SAP API Policy v4.2026a permits published APIs for documented integration and extension uses, while agentic API access must use an SAP-endorsed architecture or service-specific path. SAP’s current guidance names A2A and MCP patterns; neither bypasses the underlying API, identity, or entitlement boundary.

Worked example — Accounting Accruals Agent

When local accrual rules and manual history matter, use customer AI to prepare the proposal and SAP to control the accounting outcome.

01

Validate availability before committing to the SAP-native lane. Product, release, region, entitlement, tenant configuration, and supported interfaces can change the route.

Recommended deployment

Build the customer logic; keep accounting action controlled in SAP

The custom lane matches approved manual-accrual precedents with current evidence and prepares a traceable draft. The SAP-native lane handles packaged capability or any action that cannot be supportably exposed.

Hybrid AI Workspace

Customer AI prepares the decision

  • Match approved precedents from manual-accrual history and local policy.
  • Cross-check PO, receipt, service-entry, invoice, and contract evidence.
  • Return a constrained draft with evidence IDs, rationale, and confidence.

Deterministic services and SAP control the outcome

  • Calculate official values, coding, currency, ledger, period, and tolerances.
  • Enforce duplicate checks, segregation of duties, approval, and posting rights.
  • Post, reconcile, reverse, and close only through a permitted SAP action.

Assumptions to replace before a quote

Tasks, actions per task, tokens per task, constant FX, constant pricing, and flat three-year volume. Confirm the feature in SAP’s AI Features Guide Price List, AI Estimator, tenant, and contract.

What the figures exclude

Implementation, entitlements, hosting, SAP integration, security, cached tokens, retries, testing, evaluation, support, change, and operational ownership. SAP AI Units are not model tokens; no official conversion exists.

Commercial continuity and platform dependency

Both SAP Business AI and our managed custom platform are continuing services—not one-time development purchases. Customer data, platform intellectual property, operating responsibilities, transition support, and exit rights must be defined in the contract.

Define ownership explicitly

The customer owns its business data. Rights to platform IP, agent logic, customer configuration, and transition support depend on the chosen package and contract.

Keep SAP calls isolated

One governed integration layer should own APIs, credentials, validation, retries, and lifecycle monitoring.

Keep the manual route alive

The finance team must still be able to review, post, reconcile, and reverse without the external agent.
